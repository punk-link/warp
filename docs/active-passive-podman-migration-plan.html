Warp Infrastructure Simplification

Active/Passive Podman Migration Plan

This document reframes the earlier draft into an execution-ready plan for moving Warp from Kubernetes, Consul, and Vault KV to a simpler Podman-based deployment. It keeps dedicated servers, preserves the application architecture, starts with a single production node, and keeps active/passive as a later expansion step if usage grows enough to justify it.

Executive Summary

The original conclusion was broadly right: Kubernetes, Consul, and Vault KV add more complexity than value for this application. Warp is a monolithic web app with in-process workers, HPA is disabled, and the operational footprint can be reduced substantially by moving first to one dedicated server using Podman, systemd, and Caddy. The important corrections are in the cutover mechanics, especially around encryption format changes, the fact that passive-node worker behavior only matters once a second node exists, and the fact that GitHub Actions is the current CI/CD surface.

Recommended end state: Keep S3, GuardDuty, OpenAI, and Sentry as they are. Replace Kubernetes with Podman plus Quadlets, replace Consul with standard ASP.NET Core configuration sources, remove Vault KV, and replace Vault Transit with envelope encryption using AWS KMS only to unwrap a local AES data key at startup. Roll out on one node first and add a passive host only when traffic justifies the extra operational complexity.

Corrections to the Draft

These are the ambiguous or misleading parts that needed to be tightened before the plan could be treated as executable.

1. Direct KMS per payload is the wrong model

The application encrypts and decrypts data inside the KeyDB access path. Moving every payload through AWS KMS Encrypt and Decrypt would shift a hot path to a network-bound service, increasing latency and request cost. The correct replacement for Vault Transit here is envelope encryption: unwrap a single AES data key at startup, then encrypt payloads locally.

Do not call KMS on every KeyDB read/write Use KMS to unwrap an AES-256 key once

2. Ciphertext versioning still changes, but migration can be skipped

The post-Vault scheme still needs its own explicit format version so the application does not confuse old Vault Transit payloads with the new local-encryption format. Because production usage is currently low, the default plan can be a clean cutover during a maintenance window instead of building dual-read logic and a re-encryption utility.

Prefer clean cutover over migration by default New format version is still mandatory

3. Vault is wired into startup, not just deployment

Production startup currently uses Vault to fetch bootstrap secrets and then loads the main runtime configuration from Consul. DI also registers a Vault client unconditionally today. That means removing Vault requires code changes in startup and dependency registration, not just container changes.

Startup refactor required

4. Active/passive changes worker behavior, but it is not phase one

Warp runs background workers in the same web process. That matters once a passive node exists, because the standby app must not run duplicate cleanup or processing roles by accident. It does not need to block the initial migration, because the rollout can start with a single production node and defer passive-node worker policy until scale requires a second host.

Single-node first avoids worker collisions Define passive role before adding node two

5. GitHub Actions is the current CI/CD path

The repository contains GitHub Actions workflows that build, push, and deploy the current Helm-based environments. The migration plan therefore needs to adapt the existing GitHub Actions deployment path away from Kubernetes and Helm rather than inventing a Jenkins-centered rollout.

Replace Helm-oriented GitHub Actions steps No Jenkins dependency in the target plan

6. Podman is a fit, but systemd matters

Even for the initial dedicated server, raw compose files are not enough as the operational model. The production control plane should be Podman plus systemd Quadlets, with Caddy on the host. That gives you clean boot behavior, restart policy integration, and a predictable service boundary without rebuilding a mini-cluster.

Podman plus Quadlets plus Caddy

Initial Node

  • Runs Caddy on the host if traffic lands here.
  • Runs Warp in Podman with the worker profile enabled.
  • Runs KeyDB as the writable primary.
  • Uses host-injected secrets and production appsettings.

Passive Node Later

  • Is not required for the first production rollout.
  • Can run KeyDB as a replica of the initial node when capacity or resilience needs increase.
  • Must not run the same worker profile unless that is explicitly made safe.
  • Can keep the web app stopped until failover, or run a no-workers profile.

Configuration and Secrets

  • Move non-sensitive settings into appsettings.Production.json.
  • Inject secrets via environment files or host secret injection.
  • Remove Consul and Vault KV from the runtime path.

Encryption

  • Use AWS KMS only to unwrap a locally used AES data key at startup.
  • Keep encryption in-process for KeyDB payload performance.
  • Change the ciphertext version and prefer a low-volume maintenance cutover over a migration utility.
Operational posture: start with one node. If growth later justifies a second host, start with manual failover or provider-side IP switch and add automatic VIP movement only if manual promotion becomes a real operational burden.

Migration Phases

The sequence below is ordered to minimize coupled changes and avoid mixing infrastructure simplification with encryption cutover in a single risky step.

Flatten runtime configuration

Export the Consul KV tree for the real environments, separate sensitive and non-sensitive values, move non-secrets into appsettings.Production.json, and shift secrets to host-injected environment values. After that, remove the Consul configuration provider from startup.

Remove Vault from startup bootstrap

Decouple Vault from the application boot path by removing the production Vault bootstrap for Consul and S3 secret injection. Keep Vault code only where still needed for the temporary migration path or E2E.

Introduce envelope encryption

Add a new encryption mode that uses AWS KMS to unwrap a stored AES key at startup, then performs local symmetric encryption for KeyDB payloads. This replaces Vault Transit without pushing every record operation to AWS.

Add a new ciphertext version

Make the post-Vault format distinguishable at read time. New writes must use the new version, and the production cutover can assume a clean break instead of carrying dual-read compatibility by default.

Plan the low-volume production cutover

Because production usage is currently low, prefer a maintenance window that accepts legacy encrypted payloads expiring or being discarded rather than building a one-time KeyDB migration utility. Revisit migration tooling only if production usage materially increases before rollout.

Run the new stack on one production node

Deploy one Podman-managed production node with the normal worker profile enabled. Do not introduce passive-node runtime branching until a second host is actually needed.

Move to Podman plus systemd

Convert the production deployment to Podman-managed services, define systemd units or Quadlets, keep Caddy on the host, and carry forward only the resource controls that still matter.

Add KeyDB active/passive failover later

If traffic later justifies a second server, run a primary on the main host and a replica on the passive host. Start with a documented manual promotion procedure and only add extra coordination if it proves necessary.

Adapt GitHub Actions deployment automation

Replace the current GitHub Actions Helm deploy steps with the Podman-host deployment flow. If a second host is added later, update the passive host first and then the primary host.

Right-size observability separately

The Loki, Tempo, Prometheus, and Grafana stack can remain on Podman, but it should be evaluated as a separate cost-reduction track rather than bundled into the control-plane migration.

Primary Risks and Controls

Encryption cutover risk

A broken format cutover or a mistaken assumption about low-value legacy data would affect core data access.

  • Introduce explicit format versioning.
  • Use a maintenance window with clear handling for legacy encrypted payloads.
  • Dry-run the new write path before cutover.

Future passive-node worker collisions

Running the same hosted services on both nodes later could cause duplicate cleanup behavior or unexpected cross-node side effects.

  • Avoid the problem entirely by starting single-node.
  • Keep the standby app cold, or add a no-workers role before node two goes live.

Config drift during Consul removal

Missing or misclassified settings could break startup or subtle runtime features.

  • Inventory the Consul tree first.
  • Validate production startup before infrastructure cutover.

Hidden operational dependence on Kubernetes

Probes, restarts, and TLS termination must all be re-owned explicitly by the host runtime.

  • Use Quadlets or systemd units for restart policy.
  • Use Caddy for TLS and reverse proxy.
  • Recreate health checks and deploy sequencing.

Validation Gates

  1. Production startup succeeds without Consul and without Vault KV bootstrap, using only appsettings.Production.json and environment values.
  2. The new encryption service writes the new ciphertext version and the production cutover plan for legacy Vault Transit payloads is explicitly documented.
  3. The initial Podman deployment is verified on one production node with reads, writes, image flows, and background work behaving normally.
  4. If and when a passive node is added, its runtime is verified to keep workers disabled or the app stopped before failover.
  5. If and when a passive node is added, failover is rehearsed end-to-end: promote KeyDB replica, start or activate the passive app role, switch traffic, and verify reads, writes, image flows, and background work.
  6. The container image no longer embeds a Vault token and GitHub Actions performs the explicit deployment sequence for the new Podman-based runtime.
