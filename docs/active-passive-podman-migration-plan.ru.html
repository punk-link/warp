Упрощение инфраструктуры Warp

План миграции Active/Passive на Podman

Этот документ переформулирует предыдущий черновик в готовый к выполнению план переноса Warp из Kubernetes, Consul и Vault KV на более простую схему развертывания на базе Podman. План сохраняет выделенные серверы, архитектуру приложения, начинается с одного производственного узла и оставляет Active/Passive как последующий этап расширения, если рост нагрузки оправдает его внедрение.

Краткое изложение

Исходный вывод был в целом верным: Kubernetes, Consul и Vault KV вносят больше сложности, чем приносят пользы для данного приложения. Warp представляет собой монолитное веб-приложение с внутрипроцессными воркерами, HPA отключен, а операционный след может быть существенно уменьшен путем перехода вначале на один выделенный сервер с использованием Podman, systemd и Caddy. Важные исправления касаются механики переключения, особенно изменений формата шифрования, того факта, что поведение воркера на пассивном узле имеет значение только после появления второго узла, и того, что GitHub Actions является текущим инструментом CI/CD.

Рекомендуемое конечное состояние: Оставить S3, GuardDuty, OpenAI и Sentry без изменений. Заменить Kubernetes на Podman с Quadlets, заменить Consul стандартными источниками конфигурации ASP.NET Core, удалить Vault KV и заменить Vault Transit конвертным шифрованием (Envelope Encryption) с использованием AWS KMS только для развертывания локального AES-ключа данных при запуске. Развертывание сначала на одном узле, добавление пассивного хоста только тогда, когда трафик оправдает дополнительные операционные расходы.

Исправления к черновику

Это неоднозначные или вводящие в заблуждение части, которые необходимо было уточнить, прежде чем считать план готовым к исполнению.

1. Прямой KMS для каждой порции данных — неверная модель

Приложение шифрует и дешифрует данные в процессе доступа к KeyDB. Пропуск каждой порции данных через AWS KMS Encrypt и Decrypt превратит критический путь в зависимый от сети сервис, увеличивая задержку и стоимость запросов. Правильной заменой Vault Transit здесь является конвертное шифрование: развертывание одного AES-ключа данных при запуске и последующее локальное шифрование.

Не вызывать KMS при каждой записи/чтении KeyDB Использовать KMS для развертывания AES-256 ключа один раз

2. Версионирование шифротекста меняется, но миграцию можно пропустить

Схема после удаления Vault по-прежнему требует собственной явной версии формата, чтобы приложение не путало старые данные Vault Transit с новым форматом локального шифрования. Поскольку текущее использование в рабочей среде невелико, стандартным планом может быть чистое переключение во время окна обслуживания вместо разработки логики двойного чтения и утилиты перешифрования.

Предпочтение чистому переключению перед миграцией Новая версия формата обязательна

3. Vault задействован в запуске, а не только в развертывании

Запуск в рабочей среде в настоящее время использует Vault для получения секретов начальной загрузки, после чего загружается основная конфигурация из Consul. DI также регистрирует клиент Vault безусловно. Это означает, что удаление Vault требует изменений в коде запуска и регистрации зависимостей, а не только изменений в контейнерах.

Требуется рефакторинг процесса запуска (Startup)

4. Active/Passive меняет поведение воркеров, но это не первая фаза

Warp запускает фоновые воркеры в том же веб-процессе. Это становится важным при появлении пассивного узла, так как резервное приложение не должно случайно дублировать задачи по очистке или обработке. Это не должно блокировать первоначальную миграцию, так как развертывание можно начать с одного узла и отложить политику воркеров на пассивном узле до момента, когда потребуется второй хост.

Один узел на старте исключает коллизии воркеров Определить роль пассивного узла до добавления второго хоста

5. GitHub Actions — текущий путь CI/CD

Репозиторий содержит рабочие процессы GitHub Actions, которые собирают, отправляют и развертывают текущие среды на базе Helm. Следовательно, план миграции должен адаптировать существующий путь развертывания GitHub Actions, отойдя от Kubernetes и Helm, а не внедрять развертывание через Jenkins.

Заменить шаги GitHub Actions, ориентированные на Helm Отсутствие зависимости от Jenkins в целевом плане

6. Podman подходит, но важен systemd

Даже для первоначального выделенного сервера обычных compose-файлов недостаточно для операционной модели. Продуктивной средой управления должен быть Podman плюс systemd Quadlets с Caddy на хосте. Это обеспечивает чистое поведение при загрузке, интеграцию политик перезапуска и предсказуемые границы сервисов без необходимости создания мини-кластера.

Podman плюс Quadlets плюс Caddy

Целевая топология

Развертывание начинается с одного производственного сервера, что немедленно устраняет сложность оркестрации кластера. Второй пассивный сервер остается опциональным шагом в будущем.

Начальный узел

  • Запускает Caddy на хосте для приема трафика.
  • Запускает Warp в Podman с включенным профилем воркера.
  • Запускает KeyDB как основной узел с возможностью записи.
  • Использует секреты, внедренные на хосте, и производственные конфиги.

Пассивный узел (позже)

  • Не требуется для первого этапа развертывания.
  • Может запускать KeyDB как реплику при росте потребности в отказоустойчивости.
  • Не должен запускать тот же профиль воркера без обеспечения безопасности.
  • Может держать веб-приложение остановленным до момента отказа или использовать профиль без воркеров.

Конфигурация и секреты

  • Перенос нечувствительных настроек в appsettings.Production.json.
  • Внедрение секретов через файлы окружения или инъекцию на уровне хоста.
  • Удаление Consul и Vault KV из процесса выполнения.

Шифрование

  • Использование AWS KMS только для дешифрования локально используемого AES-ключа при запуске.
  • Сохранение шифрования внутри процесса для обеспечения производительности KeyDB.
  • Смена версии шифротекста; предпочтение окна обслуживания для переключения системы.
Операционный подход: начать с одного узла. Если рост нагрузки оправдает второй хост, использовать ручное переключение или смену IP на стороне провайдера, и добавлять автоматический VIP только если ручное продвижение станет реальным бременем.

Этапы миграции

Последовательность ниже выстроена так, чтобы минимизировать связанные изменения и избежать смешивания упрощения инфраструктуры с переключением шифрования в один рискованный шаг.

Уплощение конфигурации среды

Экспорт дерева Consul KV для реальных сред, разделение чувствительных и нечувствительных данных, перенос несекретных данных в appsettings.Production.json и переход к секретам через переменные окружения на хосте. После этого удаление провайдера Consul при запуске.

Удаление Vault из этапа запуска

Отделение Vault от пути загрузки приложения путем удаления механизма получения секретов для Consul и S3 через Vault. Сохранение кода Vault только там, где это необходимо для временного пути миграции или E2E-тестов.

Внедрение конвертного шифрования

Добавление нового режима шифрования, использующего AWS KMS для развертывания сохраненного ключа AES при запуске с последующим локальным симметричным шифрованием для KeyDB. Это заменяет Vault Transit без отправки каждой операции с записью в AWS.

Введение новой версии шифротекста

Обеспечение различимости нового формата (после удаления Vault) при чтении. Новые записи должны использовать новую версию, а переключение в рабочей среде предполагает "чистый разрыв" без поддержки обратной совместимости по умолчанию.

Планирование переключения при низкой нагрузке

Учитывая текущую низкую нагрузку, предпочтительно использовать окно обслуживания, при котором старые зашифрованные данные могут истечь или быть отброшены, вместо создания разовой утилиты миграции KeyDB. Вернуться к инструментам миграции только при существенном росте использования системы.

Запуск нового стека на одном узле

Развертывание одного производственного узла под управлением Podman с включенным стандартным профилем воркера. Не вводить ветвление логики для пассивного узла до тех пор, пока второй хост не потребуется фактически.

Переход на Podman и systemd

Конвертация деплоя в сервисы под управлением Podman, определение юнитов systemd или Quadlets, использование Caddy на хосте и перенос только тех ограничений ресурсов, которые все еще важны.

Добавление Active/Passive для KeyDB позже

Если нагрузка позже оправдает второй сервер, запустить основной узел на главном хосте и реплику на пассивном. Начать с документированной процедуры ручного продвижения реплики и добавлять координацию только при необходимости.

Адаптация автоматизации деплоя GitHub Actions

Замена текущих шагов Helm-деплоя в GitHub Actions на поток развертывания для Podman-хоста. При добавлении второго хоста сначала обновлять пассивный, затем основной.

Оптимизация мониторинга отдельно

Стек Loki, Tempo, Prometheus и Grafana может оставаться на Podman, но его следует оценивать как отдельный трек по снижению затрат, а не смешивать с миграцией уровня управления.

Основные риски и меры контроля

Риск переключения шифрования

Нарушение формата переключения или ошибочное предположение о низкой ценности старых данных может повлиять на доступ к основным данным.

  • Внедрение явного версионирования формата.
  • Использование окна обслуживания с четким регламентом обработки старых зашифрованных данных.
  • Тестовый запуск новой схемы записи перед окончательным переключением.

Коллизии воркеров на пассивном узле в будущем

Запуск одинаковых сервисов на обоих узлах в будущем может привести к дублирующей очистке или неожиданным побочным эффектам.

  • Полное избегание проблемы путем старта с одного узла.
  • Держать резервное приложение в "холодном" состоянии или добавить роль без воркеров до ввода второго узла в строй.

Расхождение конфигураций при удалении Consul

Пропущенные или неверно классифицированные настройки могут нарушить запуск или тонкие функции системы.

  • Предварительная инвентаризация дерева Consul.
  • Валидация запуска производственной среды до переключения инфраструктуры.

Скрытая операционная зависимость от Kubernetes

Проверки состояния (probes), перезапуски и терминация TLS должны быть явно перенесены на уровень хоста.

  • Использование Quadlets или юнитов systemd для политик перезапуска.
  • Использование Caddy для TLS и обратного прокси.
  • Воссоздание проверок здоровья и последовательности развертывания.

Контрольные точки валидации

  1. Запуск рабочей среды проходит успешно без Consul и без инициализации Vault KV, используя только appsettings.Production.json и переменные окружения.
  2. Новый сервис шифрования записывает данные версии с новым форматом, а план переключения для старых данных Vault Transit явно задокументирован.
  3. Первоначальное развертывание в Podman верифицировано на одном производственном узле: чтение, запись, потоки изображений и фоновая работа выполняются штатно.
  4. При добавлении пассивного узла верифицировано, что воркеры на нем отключены или приложение остановлено до момента переключения.
  5. При добавлении пассивного узла процедура переключения отработана полностью: продвижение реплики KeyDB, активация пассивного приложения, переключение трафика и проверка всех функций.
  6. Образ контейнера больше не содержит токен Vault, а GitHub Actions выполняет явную последовательность развертывания для новой среды на базе Podman.

План опирается на текущий код и файлы конфигурации, а не на общие советы по миграции.

Реальность CI/CD

GitHub Actions — это текущий путь автоматизации, который уже собирает и развертывает среды на базе Helm. Устаревший файл Jenkins не является приоритетным для оптимизации.

.github/workflows/image.yml Jenkinsfile/cicd.jenkins

Подготовлено как автономный документ, который можно просматривать непосредственно в браузере, передавать внутри команды или прикреплять к обсуждениям миграции без внешних ресурсов или рендеринга markdown.

Рекомендуемая последовательность: уплотнение конфигурации, отделение Vault от запуска, добавление нового формата шифрования, определение плана переключения, перенос среды на Podman и systemd на одном узле, а затем адаптация GitHub Actions. Политика для пассивного узла вводится только при фактической необходимости второго хоста.
